Configuration & Setup

15% Trailhead ↗

Company Information

  • Org ID — unique 15-char identifier, needed for support
  • Default Time Zone, Language, Currency
  • Data & File Storage limits
  • License counts and API usage

Licenses

  • User License — baseline features, every user must have one
  • Feature License — grants additional features (e.g. Marketing User)
  • Permission Set License — gradually grants features not in user license

Currency

  • Multi-Currency must be enabled; CANNOT be reversed
  • Advanced Currency Management = dated exchange rates for historical accuracy

Fiscal Year

  • Standard (Gregorian) or Custom (non-standard)
  • Custom Fiscal Year cannot be reverted to Standard

User Management

  • Users can never be deleted — deactivate instead
  • Freeze = blocks login, keeps license | Deactivate = blocks login + frees license
  • Login History: 20,000 records, last 6 months
  • Trusted IP Ranges bypass Device Activation prompts

Security Health Check

  • Setup → Security → Health Check
  • Scores: 90–100% Excellent | 80–89% Good | 70–79% Fair | Below 70% Poor
  • Risk categories: High, Medium, Low, Informational (Informational = not scored)
  • Compare against Salesforce Standard Baseline or upload a Custom Baseline (.xml)
  • Has inline "Fix It" buttons to resolve issues directly from the page
  • Does NOT auto-apply changes — admin must confirm each fix

Login Access Policies

  • Admin Login as Any User — enabled via Setup → Login Access Policies; logs to Setup Audit Trail
  • Grant Login Access — users voluntarily grant access to Salesforce Support or admins (User Settings → Grant Account Login Access)
  • Access can be granted for durations: 1 day, 3 days, 1 week, 1 month

My Domain

  • Custom subdomain for your Salesforce org (e.g. mycompany.my.salesforce.com)
  • Required for Lightning Experience, SSO, and some integrations
  • Visualforce page URLs change when My Domain is enabled — update any hardcoded links

  • Request can be cancelled before deploying (click "Cancel New Domain")
  • Orgs created Winter '21+ have My Domain enabled by default

Object Manager & Lightning App Builder

15% Trailhead ↗
▼

Relationship Types

LookupMaster-Detail
Child without parent?✅ Yes❌ No
Cascading delete?❌ No✅ Yes
Own sharing rules?✅ Yes❌ Inherits
Roll-Up Summary?❌ No✅ Yes

Special Relationships

  • Hierarchical — Only on User object (e.g. Manager field)
  • Self Relationship — Lookup to same object (e.g. Parent Account)
  • Many-to-Many — Junction Object with two Master-Details

Roll-Up Summary

  • Created on Master (Parent), aggregates Detail (Child) records
  • Functions: COUNT, SUM, MIN, MAX
  • Master-Detail relationship REQUIRED

Record Types

  • Controls: picklist values, page layouts, business processes
  • Assigned to Profiles (not roles)
  • Does NOT affect sharing settings

Business Processes (LOCS)

  • Sales Process → Opportunity (Stage)
  • Lead Process → Lead (Status)
  • Support Process → Case (Status)
  • Solution Process → Solution (Status)

Path

  • Visual stage guidance bar — Lightning only
  • Based on any picklist field
  • Must be added to page via Lightning App Builder

Picklists

  • Global Value Sets — defined once, reused everywhere
  • Common/Reusable Picklists do NOT exist
  • Master Picklist = used with Record Types only

Picklist Types

  • Standard Picklist — single selection
  • Multi-Select Picklist — multiple values stored with semicolons; NOT supported in formulas, rollups, or reports
  • Dependent Picklist — values filtered by a controlling field

Dependent Picklists

  • Controlling field = determines which values appear in dependent picklist
  • Controlling field must be a Picklist or Checkbox
  • Example: Country (controlling) → State (dependent)
  • Controlling field ≠ Controlled by Parent — completely different concepts (fields vs record access)

Page Layout vs Lightning Record Page

Page LayoutLightning Record Page
Built inObject ManagerLightning App Builder
ControlsFields, related lists, buttonsComponents, tabs, visibility
Assigned toProfile + Record TypeApp, Profile, or Record Type
Dynamic Forms❌ No✅ Yes (custom objects)

Dynamic Forms

  • Move fields out of Page Layout into Lightning App Builder components
  • Apply visibility rules per field or section (e.g. show only when Status = Active)
  • Available on custom objects; limited standard object support
  • Configured in Lightning App Builder — not Object Manager

Field Types — Key Distinctions

  • Formula — Read-only, calculated at runtime, cross-object lookups allowed
  • Roll-Up Summary — On Master only; aggregates child records (COUNT / SUM / MIN / MAX)
  • Validation Rule — Prevents save when formula = TRUE; runs on API & Flow too
  • Required on Page Layout ≠ Required at database level
  • Field-level Required (via FLS/Profile) = enforced on ALL saves from any interface

What can each tool control?

ToolCan control

Lightning App BuilderComponents, dynamic visibility, tabs, sidebar

Formula / Cross-Object / Roll-Up — Decision Tree

ScenarioUse
Calculate between fields on the SAME recordFormula Field
Pull a field value from a PARENT recordCross-Object Formula
Aggregate MULTIPLE child records (SUM, COUNT, MIN, MAX)Roll-Up Summary
  • Roll-Up Summary requires Master-Detail relationship — not available on Lookup
  • Roll-Up Summary is not a formula — it is its own field type
  • Validation Rules use formula syntax but are not Formula Fields
  • Multi-Select Picklist fields cannot be used in Roll-Up Summary or Formula fields

Field History Tracking

  • Tracks old value, new value, who changed it, and when
  • Max 20 fields per object
  • Retention: 18 months in UI | 24 months via API
  • Enable per object in Setup → Object Manager → [Object] → Fields → Set History Tracking

Flow Types

  • Screen Flow — UI for data entry (uses Screen element)
  • Record-Triggered — Runs on create/update/delete
  • Schedule-Triggered — Runs at a set date/time
  • Auto-Launched — Called from other automation

Key Flow Facts

  • Screen element = collects user input (has UI)
  • Assignment element = sets variables (no UI)
  • Activating a flow does NOT retroactively process existing records
  • Workflow Rules and Process Builder reached end-of-support in December 2025 — Flow is the only supported automation tool

Auto-Response Rules

  • Emails the CUSTOMER, not the record owner
  • For internal emails, use Flow

Escalation Rules

  • Can reassign or notify up to 5 email addresses
  • Time calculated using Business Hours

Approval Process

  • Submits a record for formal approval — routes to approvers, locks the record
  • Includes: Entry criteria, Approval steps, Initial/Final/Rejection actions
  • Can: lock records, send email alerts, update fields, create tasks
  • Keyword cues: "manager sign-off", "formal approval", "record lock" → Approval Process

Flow Debug — Tricky Facts

  • Active flows (screen, record-triggered, scheduled, auto-launched) save failed interviews and send a debug link in the error email
  • Platform Event–triggered flows do NOT save failed interviews → no debug link in email
  • Flows with status Draft or InvalidDraft also do NOT save failed interviews

OWD Settings

  • Private — Users see only records they own
  • Public Read Only — All can see, only owners can edit
  • Public Read/Write — All can see and edit
  • Controlled by Parent — Child record access inherits from parent record (common for Contact → Account, Case → Account)
  • Sharing Rules can NEVER be more restrictive than OWD
  • "Users need access to child records based on parent ownership" → Controlled by Parent

Public Groups Can Include

  • Can include: Users, Roles, Territories, Other Public Groups
  • Cannot include: Profiles, Permission Sets

Key Access Principles

  • Profile = object access (CAN they see Accounts?)
  • OWD + Role Hierarchy = record access (WHICH accounts?)
  • FLS controls field-level visibility and editability
  • Record Types do NOT affect sharing

Public Groups

  • Can include: Users, Roles, Roles + Subordinates, Other Public Groups, Territories
  • Cannot include: Profiles, Permission Sets
  • Max 2,000 members per group
  • Used for: Sharing Rules, Queue membership, Folder access
  • A Public Group can contain roles but is NOT itself a role

Agentforce & Einstein AI

Agentforce

  • Atlas Reasoning Engine — AI brain using ReAct (Reason + Act) prompting
  • Topics — Agent intents/specialties (e.g., 'Order Management')
  • Actions — Tools the agent uses (Flows, Apex, Prompt Templates)

  • Zero Retention — LLM does NOT store your data

Key Components of an Agent

  • Role — The agent's purpose; defines what job it does and broader goals
  • Knowledge — Data it uses: CRM records, Knowledge Articles, external data (Data Cloud)
  • Actions — Tasks it can perform: Flow, Apex, APIs, Prompt Templates
  • Guardrails — Natural-language rules for what it can/cannot do; when to escalate to a human
  • Channels — Where it operates: CRM, website, Slack, mobile, messaging

How an Agent Works

  1. Receives a trigger (user input, data change, or automation)
  2. Analyses context using the LLM
  3. Selects a topic (classifies intent)
  4. Chooses and chains actions (Flow / Apex / API)
  5. Follows guardrails and Einstein Trust Layer security
  6. Escalates to a human agent if needed

Reasoning Engine

  • Multi-turn chat — remembers conversational context across turns
  • Topic Classification — identifies user intent from natural language
  • Knowledge Retrieval (RAG) — retrieves relevant data chunks for accurate answers
  • Dynamic decision-making — selects and chains actions automatically
  • Can access public data securely via the Einstein Trust Layer

Out-of-the-Box Agents

  • Service Agent — replaces chatbots; handles service issues without scripted scenarios
  • SDR (Sales Dev Rep) — engages leads 24/7, answers questions, schedules meetings
  • Sales Coach — AI role-play training for sales reps using deal data
  • Campaign Optimizer — automates full marketing campaign lifecycle
  • Buyer Agent / Personal Shopper — ecommerce buying assistance
  • Merchandiser — assists ecommerce site setup and promotions

Quick Reference Cheat Sheet

Key facts that commonly appear in exam questions

"CANNOT be reversed"

  • Enabling Multi-Currency
  • Enabling Custom Fiscal Year

Common Traps

  • Users can NEVER be deleted
  • Auto-Response emails the CUSTOMER, not owner
  • Record Types do NOT affect sharing
  • Roll-Up Summary needs Master-Detail (not Lookup)
  • Sharing Rules can NEVER be stricter than OWD
  • Multi-Select Picklist CANNOT be used in formulas, reports, or rollups
  • Controlling Field (picklist dependency) ≠ Controlled by Parent (OWD)
  • Platform Event flows do NOT save failed interview links
  • App Manager (admin) ≠ App Launcher (end user)

Key Limits

  • Import Wizard: 50,000 records max
  • Data Loader: 150 million records max
  • Login History: 20,000 records, 6 months
  • Field History Tracking: 20 fields per object | 18 months UI | 24 months API
  • Bucket Fields: 5 per report, 20 buckets each
  • External IDs: up to 25 per object
  • Recycle Bin: auto-deletes after 15 days
  • Password length: 5–50 characters (not 3–100)

Order of Execution

  1. Validation Rules
  2. Duplicate Rules
  3. Before-Save Flows
  4. After-Save Flows
  5. Assignment Rules
  6. Auto-Response Rules
  7. Workflow Rules
  8. Escalation Rules

Access Model

  • Profile/Permission Set → Object-level access (CAN they see the object?)
  • OWD → Record baseline (most restrictive)
  • Role Hierarchy → Vertical access (manager sees subordinate's records)
  • Sharing Rules → Horizontal exceptions to OWD
  • FLS → Field-level visibility and editability

Relationship Summary

  • Lookup = loose, child optional, own sharing, no roll-up
  • Master-Detail = tight, child required, inherits sharing, roll-up enabled
  • Hierarchical = User object only
  • Standard Object must be Master side
  • Many-to-Many = Junction Object with 2 Master-Details

Report Formats

  • Tabular = no groupings
  • Summary = row groupings
  • Matrix = row + column groupings
  • Joined = multiple blocks
  • Format adapts automatically — not selected manually
  • Joined Reports always export as .xlsx

Exam Strategy Tips

  • Eliminate 2 wrong answers first — usually 2 are obviously off; you're now 50/50
  • Pick the most standard / least complex solution
  • Salesforce favours declarative (Flow) over code (Apex)
  • "Without code" = Flow | "Schedule" = Scheduled Flow | "UI" = Screen Flow
  • Flag and move on — don't spend more than 90 seconds per question
  • You need ~65% (40/60 questions) to pass — not perfection
  • Watch for double negatives ("which does NOT…")

Agentforce Key Concepts

  • Agent = autonomous AI — not just a chatbot; proactive & self-directed
  • Uses LLMs + Salesforce CRM data + External data (Data Cloud)
  • Actions = how it does work: Flow, Apex, APIs, Prompt Templates
  • Guardrails = safety rules + escalation logic
  • Reasoning Engine = decides what topic + action to take next
  • Operates 24/7 across CRM, web, Slack, mobile, messaging
  • Can escalate to human agents when needed

5 Agent Components (ROLE-K-A-G-C)

  1. Role — purpose/job
  2. Knowledge — data sources
  3. Actions — tasks it performs
  4. Guardrails — rules & limits
  5. Channels — where it operates

Agentforce Exam Traps

  • "What defines what an agent can/cannot do?" → Guardrails
  • "What identifies the agent's intent?" → Topic Classification
  • "What makes agents smarter across a conversation?" → Multi-turn Reasoning Engine
